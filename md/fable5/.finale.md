## 2. LOT D1 — DOCKERISATION (état exact)

**Décision stratégique de la session** : abandon de Render au profit d'un **auto-hébergement sur le mini PC** de l'utilisateur (le même qui héberge déjà un serveur RustDesk sous Docker Desktop, allumé H24, résilience reboot/coupure déjà validée). Motivations : Render free tier **bloque les ports SMTP 25/465/587** (confirmé — c'est la cause racine de l'échec email en prod), cold start ~30 s, et coût. La connexion Bouygues du mini PC n'a pas ces blocages → Gmail SMTP y refonctionnera.

### Architecture cible
```
Ami/proche → internet → box Bouygues (NAT 80+443 → IP locale mini PC)
    → Caddy (conteneur, TLS Let's Encrypt auto, HTTPS)
    → joker (conteneur : Express + Socket.IO + client React buildé servi en statique)
    → Neon (Postgres, cloud — INCHANGÉ)
```

### Décisions d'infra validées avec l'utilisateur
1. **BDD reste sur Neon** (cloud). Pas de Postgres dans un conteneur (durabilité, backups, pas de migration de données).
2. **HTTPS via Caddy** en reverse proxy (certif Let's Encrypt auto-renouvelé). **Obligatoire** car le cookie de session est `secure: true` en prod → sans HTTPS, pas d'auth. Caddy = équivalent Nginx ; `trust proxy: 1` reste correct (un proxy devant Express, comme sur Render).
3. **Gmail SMTP réactivé** (ports non bloqués chez Bouygues). Variables dans l'env du conteneur.
4. **2e domaine DuckDNS** (ex. `joker-xxx.duckdns.org`) sur le compte existant + 2 règles NAT box (80 TCP, 443 TCP → IP locale mini PC).

### Fichiers créés par l'agent D1 (racine du monorepo) — VALIDÉS
- `Dockerfile` — multi-étages : `builder` (node:20) build le client (`vite build`) + installe deps prod ; runtime (`node:20-slim`) assemble `server/` + `shared/` (source .ts, tournés en tsx) + `client/dist` + node_modules prod. `WORKDIR /app/server`, `CMD ["npm","start"]`.
- `.dockerignore` — exclut node_modules, .git, .env, */dist, *.test.ts, md/, handoff.md…
- `docker-compose.yml` — services `joker` (build local, `env_file: .env`, pas de port exposé à l'hôte, `restart: unless-stopped`) + `caddy` (ports 80/443, **volumes de persistance des certs** `caddy_data`/`caddy_config`, `depends_on: joker`). Réseau bridge (⚠️ **pas** `network_mode: host` — interdit sous Docker Desktop Windows, piège connu RustDesk).
- `Caddyfile` — `{$JOKER_DOMAIN} { reverse_proxy joker:{$PORT} }`, rien de hardcodé.
- `.env.example` (racine) — clés sans valeurs : `DATABASE_URL`, `SMTP_USER`, `SMTP_PASS`, `JOKER_DOMAIN`, `CLIENT_URL`, `PORT`, `NODE_ENV`.
- `DEPLOY.md` — install pas-à-pas mini PC.

### Reconnaissance D1 (faits établis sur le code réel)
- `server/src/index.ts` **sert déjà** le client : `express.static(../client/dist)` (l.612-619) + catch-all SPA `app.get(/.*/, sendFile index.html)` (l.621-623), monté APRÈS `/api` et Socket.IO. **Non modifié** (rien à faire).
- Serveur en `tsx` (pas de build serveur). `npm start` = `tsx src/index.ts`. Doit démarrer depuis `server/` (résout `client/dist` via `process.cwd()/../client/dist`).
- `PORT` lu depuis `process.env.PORT`. Cookie `secure: NODE_ENV==='production'`. `trust proxy: 1` déjà posé (H1).
- **`CLIENT_URL`** (index.ts:52) = origine CORS de Socket.IO. **Doit valoir `https://<JOKER_DOMAIN>`** en prod, sinon WebSockets refusés. (Absent du brief initial, trouvé en recon.)
